# ERROR Log Alerting with Automated GitHub Issue Creation

This document describes the alerting system for WellKorea ERP, which automatically creates GitHub issues when ERROR logs are detected.

## Architecture

```
Backend logs → Promtail → Loki → Grafana Alert Rule → Webhook → n8n → GitHub Issue
                           ↑
                   (level label extracted)
```

### Components

| Component | Role |
|-----------|------|
| **Promtail** | Ships container logs to Loki, extracts `level` and `correlationId` from JSON logs |
| **Loki** | Log aggregation with 7-day retention |
| **Grafana** | Alert rules monitor for ERROR logs, triggers webhook on detection |
| **n8n** | Workflow automation that fetches correlated logs and creates GitHub issues |
| **GitHub** | Issue tracking for error investigation |

## How It Works

1. Backend logs are shipped to Loki via Promtail
2. Grafana evaluates alert rules every 1 minute
3. When ERROR logs are detected, Grafana fires an alert to n8n webhook
4. n8n workflow:
   - Receives Grafana alert payload
   - Queries Loki for ERROR logs from the last 5 minutes
   - Extracts unique `correlationId` values from ERROR logs
   - For each correlationId, fetches the complete log chain (all log levels)
   - Creates a GitHub issue with full request context

### Correlation ID

Each HTTP request is assigned a UUID-based correlation ID (`X-Request-ID` header). All logs within that request share the same `correlationId` in MDC. This allows reconstructing the full request flow leading to an error.

Example log chain for a single request:
```
[INFO]  AuthController: Login request received for user: john@example.com
[DEBUG] AuthService: Validating credentials
[INFO]  AuthService: User authenticated successfully
[WARN]  ProjectController: Large project list requested (500+ items)
[ERROR] ProjectService: Database connection timeout while fetching projects
```

## Configuration

### Environment Variables

Add to `.env`:

```bash
# n8n Configuration
N8N_WEBHOOK_URL=http://n8n:5678/webhook/grafana-alerts
N8N_HOST=n8n.wellkorea.mipllab.com
N8N_PROTOCOL=https
N8N_WEBHOOK_BASE_URL=https://n8n.wellkorea.mipllab.com

# GitHub Integration
GITHUB_PAT=ghp_xxxxxxxxxxxx
GITHUB_OWNER=donghwunline
GITHUB_REPO=wellkorea-erp
```

### GitHub Personal Access Token

1. Go to [GitHub Fine-grained PAT settings](https://github.com/settings/personal-access-tokens/new)
2. Create a token with:
   - **Repository access**: Only select `wellkorea-erp`
   - **Permissions**: `Issues: Read and write`
3. Copy the token to `GITHUB_PAT` in `.env`

### GitHub Labels

Pre-create these labels in the `wellkorea-erp` repository:
- `automated-alert` (color: #d73a4a) - Auto-generated by monitoring
- `monitoring` (color: #0366d6) - Monitoring/observability related
- `critical` (color: #b60205) - Critical severity

## n8n Workflow Setup

After deploying the monitoring stack, configure the n8n workflow:

### 1. Access n8n

Open `https://n8n.wellkorea.mipllab.com` (or `http://192.168.0.16:20050` internally)

### 2. Create Webhook Trigger

1. Add a **Webhook** node
2. Set path to `grafana-alerts`
3. Method: POST
4. This creates the endpoint: `http://n8n:5678/webhook/grafana-alerts`

### 3. Parse Alert Payload

Add a **Set** node to extract:
- `alertname`: `{{ $json.alerts[0].labels.alertname }}`
- `service`: `{{ $json.alerts[0].labels.service }}`
- `severity`: `{{ $json.alerts[0].labels.severity }}`
- `startsAt`: `{{ $json.alerts[0].startsAt }}`

### 4. Fetch ERROR Logs from Loki

Add an **HTTP Request** node:
- Method: GET
- URL: `http://loki:3100/loki/api/v1/query_range`
- Query Parameters:
  - `query`: `{service="backend", level="ERROR"}`
  - `start`: `{{ Date.now() - 300000 }}000000` (5 min ago in nanoseconds)
  - `end`: `{{ Date.now() }}000000`

### 5. Extract Correlation IDs

Add a **Code** node:
```javascript
const results = $input.first().json.data.result;
const correlationIds = new Set();

for (const stream of results) {
  for (const [timestamp, logLine] of stream.values) {
    try {
      const log = JSON.parse(logLine);
      if (log.correlationId) {
        correlationIds.add(log.correlationId);
      }
    } catch (e) {
      // Skip non-JSON logs
    }
  }
}

return Array.from(correlationIds).map(id => ({ correlationId: id }));
```

### 6. Fetch Full Log Context (Loop)

For each correlation ID, add an **HTTP Request** node in a loop:
- URL: `http://loki:3100/loki/api/v1/query_range`
- Query: `{service="backend"} | correlationId="{{ $json.correlationId }}"`

### 7. Check for Duplicate Issues

Add a **GitHub** node:
- Operation: Search Issues
- Query: `repo:donghwunline/wellkorea-erp is:open label:automated-alert`

### 8. Create GitHub Issue

Add a **GitHub** node (only if no duplicate found):
- Operation: Create Issue
- Repository: `wellkorea-erp`
- Title: `[ALERT] ERROR Logs Detected - {{ $json.service }}`
- Labels: `automated-alert`, `monitoring`, `critical`
- Body: (see Issue Template below)

## Alert Configuration

### Alert Rules (monitoring/grafana/provisioning/alerting/alerts.yml)

| Setting | Value | Description |
|---------|-------|-------------|
| Query | `sum(count_over_time({service="backend", level="ERROR"}[5m]))` | Count ERROR logs in 5-min window (aggregated) |
| Query Type | instant | Returns single value for alerting |
| Threshold | > 0 | Fire when any ERROR logs exist |
| Evaluation interval | 1 minute | How often to check |
| For duration | 0s | Fire immediately when condition is met |
| No data state | OK | Don't alert when there are no logs |

**Why `sum()` wrapper?**
Grafana alerting requires a scalar or instant vector (single value), not a range vector (time series data). The `sum()` aggregation ensures all log streams are combined into one value, which Grafana can then evaluate against the threshold.

### Notification Policy (policies.yml)

| Setting | Value | Description |
|---------|-------|-------------|
| group_wait | 30s | Wait before first notification |
| group_interval | 5m | Wait before notifying about new alerts in same group |
| repeat_interval | 4h | Wait before re-notifying for same alert |

## Verification

### 1. Test Alert Firing

Generate a test ERROR log in the backend:

```java
log.error("Test error for alerting verification");
```

Or via curl if you have an endpoint that logs errors.

### 2. Verify Grafana Alert

1. Go to Grafana → Alerting → Alert rules
2. Find "ERROR Level Logs Detected"
3. Check if state changes to "Firing"

### 3. Check n8n Webhook

1. Go to n8n → Executions
2. Verify the workflow was triggered
3. Check each step completed successfully

### 4. Verify GitHub Issue

1. Go to GitHub → Issues
2. Find the auto-created issue with `automated-alert` label
3. Verify it contains the full log context

## Troubleshooting

### Alert Not Firing

1. Check if ERROR logs exist in Loki:
   ```
   {service="backend", level="ERROR"}
   ```
2. Verify Grafana datasource connection
3. Check alert rule evaluation in Grafana UI

### Webhook Not Received by n8n

1. Verify n8n is running: `docker logs wellkorea-erp-n8n`
2. Check webhook URL in Grafana contact point
3. Ensure n8n is in the same Docker network (`wellkorea-internal`)

### GitHub Issue Not Created

1. Check n8n execution logs for errors
2. Verify `GITHUB_PAT` has correct permissions
3. Test GitHub API manually:
   ```bash
   curl -H "Authorization: token $GITHUB_PAT" \
     https://api.github.com/repos/donghwunline/wellkorea-erp/issues
   ```

### Loki Query Returning Empty

1. Verify log format matches expected JSON structure
2. Check Promtail is extracting labels correctly:
   ```bash
   docker logs wellkorea-erp-promtail
   ```
3. Query without filters first: `{service="backend"}`
